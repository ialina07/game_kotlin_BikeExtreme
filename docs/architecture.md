# Архитектура BikeExtreme 

`Диаграмма классов в файле architecture.jpg`

## Описание архитектуры

### 1. Слои архитектуры

Приложение разделено на три слоя:

| Слой | Назначение | Классы |
|------|-----------|--------|
| **Domain (сущности)** | Хранение данных | Game, Player, Move, PlayerState |
| **Бизнес-логика** | Правила игры, фазы хода, валидация | Phase, WeatherPhase, EventPhase, MovementPhase, EnergyPhase, RestPhase, PhaseExecutor, PhaseContext, MoveValidator |
| **Бизнес-логика (периферия)** | Статистика и рейтинги | StatisticsService |
| **UI слой** | Ввод и вывод в консоли | ConsoleUI |
| **Хранение** | Сохранение и загрузка данных | GameRepository (интерфейс), InMemoryGameRepository |

### 2. Основные сущности (data-классы)

| Класс | Поля | Назначение |
|-------|------|-----------|
| **Game** | id, date, status | Партия игры |
| **Player** | id, name | Игрок (существует отдельно от партий) |
| **Move** | id, playerId, turnNumber, dice1, dice2, stateBefore, stateAfter, isValid | Один ход с состоянием до и после |
| **PlayerState** | position, energy, condition, water | Ресурсы игрока в данный момент |

### 3. UI слой (ConsoleUI)

**ConsoleUI** — отвечает за ввод и вывод в консоли

| Метод | Назначение |
|-------|-----------|
| start() | Запускает главный цикл приложения (показывает меню, читает команды) |
| showMenu() | Выводит список доступных команд |
| createGame() | Создать новую партию |
| inputMove() | Ввести ход |
| showGameState(gameId) | Показать текущее состояние игры |
| showPlayerStats(playerName) | Показать статистику игрока |
| showLeaderboard() | Показать таблицу лидеров |
| showReplay(gameId) | Повторить партию |
| exit() | Выход из программы |

### 4. Бизнес-логика (фазы хода)

Вместо одного большого класса, логика разбита на отдельные фазы.

**Phase** (интерфейс) — обязывает каждую фазу иметь метод execute(state, context)

| Класс | Что делает |
|-------|-----------|
| WeatherPhase | Применяет погоду (через WeatherFactory) |
| EventPhase | Применяет случайное событие (через EventFactory) |
| MovementPhase | Рассчитывает движение (dice1 + dice2 + бонус) |
| EnergyPhase | Списывает 1 энергию за ход |
| RestPhase | Применяет отдых (восстановление ресурсов) |

**PhaseExecutor** — применяет все фазы по порядку к состоянию игрока.

**PhaseContext** — контейнер данных для фаз:
- dice1 (погода)
- dice2 (событие)
- moveType (move / rest)
- restType (energy / condition / water)
- tailwindBonus (бонус от попутного ветра)

*Если понадобится новый параметр, я просто добавлю его в PhaseContext. Ни одну фазу менять не придётся*

### 5. Погода (паттерн Стратегия + Фабрика)

**Weather** (интерфейс) — обязывает каждый тип погоды иметь метод apply(state)

| Класс | Эффект |
|-------|--------|
| HeatWeather | Жара: -1 вода (или -1 энергия, если нет воды) |
| RainWeather | Дождь: -1 состояние |
| NormalWeather | Обычно: без изменений |
| TailwindWeather | Попутный ветер: +2 клетки к движению |
| StormWeather | Шторм: -1 состояние |

**WeatherFactory** — по значению dice1 возвращает нужный объект погоды.

*Если возникнет желание усложнить игру (например сделать кубик 12-гранным), достаточно поменять только фабрику*

### 6. События (паттерн Стратегия + Фабрика)

**Event** (интерфейс) — обязывает каждое событие иметь метод apply(state)

| Класс | Эффект |
|-------|--------|
| SpringEvent | Родник: +1 вода |
| PunctureEvent | Прокол колеса: -2 состояния |
| SnackEvent | Перекус: +1 энергия |
| RepairEvent | Починка: +2 состояния |
| DownhillEvent | Спуск: +2 клетки |
| SprintEvent | Спринт: +3 клетки, -1 энергия |

**EventFactory** — по значению dice2 возвращает нужное событие.

### 7. Валидация хода (MoveValidator)

**MoveValidator** — проверяет, возможен ли ход.

| Метод | Параметры | Назначение |
|-------|----------|-----------|
| validate() | move, currentPlayerId, currentSnapshots | Проверяет ход: очередь, соответствие правилам |

**Параметры:**
- move — объект хода (содержит dice1, dice2, stateBefore, stateAfter, playerId)
- currentPlayerId — кто сейчас должен ходить (проверка очереди)
- currentSnapshots — текущие ресурсы всех игроков

**Как работает:**
1. Проверяет, тот ли игрок ходит
2. Вызывает PhaseExecutor для расчёта ожидаемого состояния
3. Сравнивает ожидаемое состояние с move.stateAfter
4. Возвращает true (валиден) или false (невалиден)

### 8. Координатор игры (GameManager)

**GameManager** — координатор игры.

| Метод | Параметры | Назначение |
|-------|----------|-----------|
| startGame() | playerNames | Создать новую партию |
| recordMove() | move | Записать ход (с валидацией) |

**Что делает GameManager:**
- Хранит текущее состояние (кто ходит, ресурсы игроков)
- Вызывает MoveValidator для проверки хода
- Сохраняет валидные ходы в репозиторий
- Переключает ход на следующего игрока

### 9. Статистика (StatisticsService)

**StatisticsService** — сервис для расчёта статистики и рейтингов.

| Метод | Назначение |
|-------|-----------|
| getPlayerStats(playerId) | Количество игр, побед, win rate |
| getLeaderboard() | Таблица лидеров (список игроков, отсортированный по количеству побед) |
| getWinRate(playerId) | Процент побед (победы / всего игр * 100)|

**Хранение истории:** GameRepository хранит все игры и ходы. StatisticsService использует репозиторий для получения данных и расчёта статистики.

### 10. Повтор партий (ReplayService)

**ReplayService** — воспроизводит партию через игровой движок.

| Метод | Назначение |
|-------|-----------|
| replayGame(gameId) | Воспроизвести партию по ID |
| getMoveHistory(gameId) | Получить историю ходов партии |

**Как работает:**
1. Загружает все ходы партии из репозитория
2. Берёт начальное состояние
3. Для каждого хода применяет PhaseExecutor (тот же движок, что в игре)
4. Сравнивает результат с сохранённым состоянием после хода
5. Выводит состояние после каждого хода

*Replay использует тот же PhaseExecutor, что и основная игра, а не просто печатает ходы*

### 11. Хранение данных

**Repository:** интерфейс GameRepository отделяет бизнес-логику от способа хранения.

**InMemoryGameRepository**:
- хранит данные в списках
- данные теряются при выключении программы
- может быть заменён на SQLite

| Метод | Назначение |
|-------|-----------|
| saveGame(game) | Сохранить партию |
| getGame(id) | Получить партию по ID |
| saveMove(move) | Сохранить ход |
| getMoves(gameId) | Получить все ходы партии |
| getAllGames() | Получить все партии |
| getGamesByPlayer(playerId) | Получить партии игрока |

### 12. Поток исполнения

### Валидация одного хода

1. **Администратор вводит данные хода:**
   - игрок
   - dice1 (погода)
   - dice2 (событие)
   - тип хода (move или rest)

2. **GameManager получает ход.** У него уже есть:
   - кто сейчас ходит
   - текущие ресурсы всех игроков

3. **GameManager вызывает MoveValidator.validate(move, currentPlayerId, currentSnapshots)**

4. **MoveValidator проверяет:**
   - тот ли игрок сейчас ходит (проверка очереди)
   - вызывает PhaseExecutor для расчёта ожидаемого состояния:
     - WeatherPhase (применяет погоду через WeatherFactory)
     - EventPhase (применяет событие через EventFactory)
     - MovementPhase (рассчитывает движение)
     - EnergyPhase (списывает энергию)
   - сравнивает ожидаемое состояние с move.stateAfter

5. **MoveValidator возвращает результат:** валиден или нет

6. **GameManager получает результат валидации**

7. **Если ход валидный:**
   - сохраняет ход в GameRepository
   - обновляет текущее состояние игры
   - переключает ход на следующего игрока

8. **Если ход невалидный:**
   - не меняет состояние игры
   - возвращает сообщение об ошибке администратору
