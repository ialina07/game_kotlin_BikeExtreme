# Архитектура BikeExtreme 

`Диаграмма классов в файле architecture.jpg`

## Описание архитектуры

### 1. Слои архитектуры

Приложение разделено на три слоя:

| Слой | Назначение | Классы |
|------|-----------|--------|
| **Domain (сущности)** | Хранение данных | Game, Player, Move, PlayerState |
| **Бизнес-логика** | Правила игры и валидация | GameRules, MoveValidator, GameManager |
| **Хранение** | Сохранение и загрузка данных | GameRepository (интерфейс), InMemoryGameRepository |

### 2. Основные сущности (data-классы)

| Класс | Поля | Назначение |
|-------|------|-----------|
| **Game** | id, date, status | Партия игры |
| **Player** | id, name | Игрок (существует отдельно от партий) |
| **Move** | id, playerid, turnNumber, diceMove, diceWeather, stateBefore, stateAfter, isValid | Один ход с состоянием до и после |
| **PlayerState** | position, energy, condition, water | Ресурсы игрока в данный момент |

### 3. Бизнес-логика

**GameRules** - применяет правила игры к состоянию игрока:
- applyWeather() - жара, дождь, ветер, обычная погода, шторм
- applyMovement() - движение по выпавшему числу
- applyEnergyCost() - расход энергии за ход
- applyEvent() - случайные события (родник, прокол колеса, починка, спуск, спринт)
- applyRest() - отдых (восстановление ресурсов: +2 энергии, +1 состояние, или кинуть кубик и получить воды столько, сколько выпало)
- canRest() - проверка, есть ли что восстанавливать (энергия, состояние или вода не на максимуме)
- getWinner() - определение победителя

**MoveValidator** — проверяет, возможен ли ход:
- соответствует ли ход правилам (вызывает GameRules)
- сравнивает ожидаемое состояние с фактическим

**GameManager** — координатор игры:
- хранит текущее состояние
- вызывает валидатор при каждом ходе
- сохраняет валидные ходы в репозиторий

### 4. Хранение данных

**Repository:** интерфейс `GameRepository` отделяет бизнес-логику от способа хранения.

**InMemoryGameRepository** 
- хранит данные в списках
- данные теряются при выключении программы
- может быть заменён на SQLite

### 5. Поток исполнения (валидация хода)

### Валидация одного хода
- Администратор вводит данные хода в систему
(игрок, кубик движения, кубик погоды, кубик события, тип хода)


- GameManager получает ход и текущее состояние игры (GameState)


- GameManager вызывает MoveValidator.validate(move, currentState)


- MoveValidator проверяет:
  
~ соответствует ли ход правилам (вызывает GameRules)

~ сравнивает ожидаемое состояние с фактическим


- MoveValidator вызывает GameRules для расчёта ожидаемого состояния

GameRules.applyWeather() - применяет погоду

GameRules.applyMovement() - рассчитывает движение

GameRules.applyEnergyCost() - списывает энергию

GameRules.applyEvent() - применяет случайное событие


- MoveValidator сравнивает:

ожидаемое состояние (от GameRules)

фактическое состояние (которое ввёл администратор)


Если совпадают -> ValidationResult.VALID

Если не совпадают -> ValidationResult.INVALID(причина)


- GameManager получает результат валидации


Если ход валидный:

~ сохраняет ход в GameRepository

~ обновляет текущее состояние игры

~ переключает ход на следующего игрока


Если ход невалидный:

~ не меняет состояние игры

~ возвращает сообщение об ошибке администратору
